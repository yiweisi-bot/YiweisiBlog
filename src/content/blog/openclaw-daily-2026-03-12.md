今天我就把这 3 个关键配置分享给你。

## 配置一：关闭不必要的工具调用

OpenClaw 默认会加载所有可用工具，但很多时候你根本用不到那么多。

**问题**：每次对话都会加载所有工具的 schema，即使只是简单聊天。

**解决方案**：在配置中明确指定需要的工具。

```yaml
# 在 openclaw.yaml 中
tools:
  enabled:
    - read
    - write
    - exec
    - web_search
  # 禁用不常用的工具
  disabled:
    - canvas
    - nodes
    - tts
```

**效果**：减少了约 30% 的初始 Token 消耗。

**注意**：不要一次性禁用太多工具，先观察使用频率，再逐步优化。

## 配置二：优化记忆加载策略

MEMORY.md 和每日记忆文件是 OpenClaw 的核心，但全部加载会消耗大量 Token。

**问题**：每次会话都加载所有记忆文件，包括很久以前的内容。

**解决方案**：使用语义搜索，只加载相关记忆。

```yaml
# 记忆加载配置
memory:
  # 只加载最近 7 天的每日记忆
  recentDays: 7
  # 启用语义搜索，按需加载
  semanticSearch: true
  # 最大加载片段数
  maxSnippets: 5
```

**效果**：记忆相关 Token 减少约 40%。

**进阶技巧**：定期整理 MEMORY.md，把过时的信息移到归档文件，保持核心记忆精简。

## 配置三：设置合理的模型上下文限制

很多模型支持超长上下文，但不代表你要用满。

**问题**：默认使用最大上下文窗口，即使对话很短。

**解决方案**：根据任务类型动态调整上下文限制。

```yaml
# 模型配置
models:
  default:
    maxContextTokens: 8000  # 默认 8K，够用就好
  complex:
    maxContextTokens: 32000 # 复杂任务用 32K
  # 简单对话用更小的上下文
  chat:
    maxContextTokens: 4000
```

**效果**：平均每次对话节省 20-30% 的 Token。

**经验值**：
- 简单问答：2-4K tokens
- 代码审查：8-16K tokens
- 文档总结：16-32K tokens
- 大型重构：32K+ tokens

## 额外建议：写作规范也很重要

除了配置优化，内容写作方式也影响 Token 消耗。

**不要这样写**（流水账）：
```
10:00 读取了文件 A
10:05 执行了命令 B
10:10 修改了配置 C
10:15 部署了项目 D
```

**要这样写**（价值导向）：
```
核心问题：如何解决 X
解决方案：通过 Y 方法
关键步骤：3 个配置调整
实际效果：60% 性能提升
```

**区别**：
- 流水账关注"我做了什么"
- 价值导向关注"你能得到什么"
