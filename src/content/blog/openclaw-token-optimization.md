如果你也在用 OpenClaw，可能遇到过这个问题：明明没做多少事，Token 消耗却蹭蹭往上涨。

经过一周的实测和调优，我找到了 3 个最有效的优化点，**Token 消耗从每天 200K+ 降到了 80K 左右**。今天把完整方案分享给你。

---

## 问题发现：Token 都花在哪了？

先看一下我的使用场景：
- 日常对话问答
- 博客内容创作
- 代码审查和优化
- 定时任务（心跳、邮件检查等）

一开始我以为主要消耗在内容创作上，但分析日志后发现：**真正的大头是插件自动注入的 Policy 内容**。

每次消息发送前，skillhub 插件会自动注入一段几百字的 policy，告诉 AI"你应该怎么做"。单次对话可能感觉不到，但一天几十次对话下来，**额外消耗了 40% 的 Token**。

---

## 优化方案 1：关闭不需要的插件 Policy ⭐⭐⭐

### 问题根源

OpenClaw 的插件系统很强大，但有些插件会通过 `before_prompt_build` 事件钩子自动注入内容。这些内容对你可能完全没用，但每次都要消耗 Token。

### 解决方法

编辑配置文件 `~/.openclaw/config/openclaw.json`：

```json
{
  "skills": {
    "skillhub": {
      "enabled": false
    }
  }
}
```

然后重启网关：

```bash
openclaw gateway restart
```

### 效果

- ✅ 每次消息减少 300-500 Token
- ✅ 消息处理速度提升约 15%
- ⚠️ skillhub 相关功能将不可用（如需使用可重新启用）

**建议**：检查所有已安装插件，关闭不需要的。插件不是越多越好。

## 优化方案 2：合理配置心跳任务

### 常见误区

很多人会把心跳任务配置得很复杂：检查邮件、检查日历、检查项目状态、生成报告... 每次心跳都消耗大量 Token。

### 推荐做法

**双轨制心跳**：

1. **简化版**（HEARTBEAT.md）：系统级快速检查，2 秒内完成
   - 只确认系统正常运行
   - 不执行耗时操作
   - 每 30 分钟一次

2. **完整版**（HEARTBEAT-full.md）：深度检查，由定时任务执行
   - 邮件、日历、项目检查
   - 生成详细报告
   - 每 3 小时一次

### 配置示例

**简化版心跳**（每次心跳都执行）：
```markdown
# HEARTBEAT.md

## 检查项
1. 快速确认系统正常运行
2. 整理当前会话上下文到记忆文件

## 原则
- 2 秒内完成检查
- 如果一切正常，返回 HEARTBEAT_OK
- 不执行复杂操作
```

**完整版心跳**（定时任务执行）：
```bash
# crontab -e
0 */3 * * * /root/.openclaw/workspace/scripts/heartbeat-full.sh
```

### 效果

- ✅ 系统级心跳 Token 消耗减少 80%
- ✅ 深度检查仍然保留，但频率降低
- ✅ 总体 Token 消耗减少约 25%

## 优化方案 3：模型选择策略

### 常见误区

- 所有任务都用最大模型（上下文越大越好）
- 不看任务复杂度，一律用顶级模型
- 没有配置备用模型，主模型失败就报错

### 推荐策略

**按任务分配模型**：

| 任务类型 | 推荐模型 | 原因 |
|---------|---------|------|
| 日常对话 | 豆包 ark-code-latest | 128K 上下文，性价比高 |
| 代码审查 | 智谱 GLM-5 | 200K 上下文，适合长代码 |
| 简单查询 | DeepSeek Chat | 响应快，成本低 |
| 长文档总结 | 智谱 GLM-5 | 200K 上下文优势 |

**配置示例**：

```json
{
  "agents": {
    "main": {
      "model": "doubao/ark-code-latest",
      "fallbackModels": ["zhipu/glm-5", "deepseek/deepseek-chat"]
    },
    "dev": {
      "model": "zhipu/glm-5",
      "fallbackModels": ["zhipu/glm-4.7", "deepseek/deepseek-chat"]
    }
  }
}
```

### 效果

- ✅ 日常对话成本降低 30%
- ✅ 复杂任务仍有大模型支持
- ✅ 主模型失败时自动切换，不中断任务

## 额外建议：3 个容易被忽视的细节

### 1. 定期清理记忆文件

记忆文件会累积，每次对话都可能被作为上下文发送。建议：
- 每周清理一次过期的 daily memory
- MEMORY.md 保持精炼，只保留核心信息
- 使用向量数据库存储历史记忆（如 Agent Memory Ultimate）

### 2. 避免在公开内容中泄露配置

这是用血换来的教训：
- ❌ 不要在博客中写具体的数据库路径
- ❌ 不要暴露 API 端口和内部 IP
- ❌ 不要提交配置文件到 GitHub

**正确做法**：
- 用占位符代替敏感信息：`<YOUR_API_KEY>`
- 配置文件加入 `.gitignore`
- 发布前用安全扫描工具检查

### 3. 监控和告警

配置 Token 消耗监控：
- 设置每日预算上限
- 超出阈值时自动告警
- 每周生成消耗报告，分析趋势

## 总结

Token 优化不是一次性工作，而是持续的过程。核心原则：

1. **按需配置**：不需要的功能就关闭
2. **分级处理**：简单任务用小模型，复杂任务用大模型
3. **定期检查**：每周查看日志，发现新的优化点

**最后提醒**：优化不是为了省钱而省钱，而是为了让资源用在刀刃上。该花的地方还是要花，不该花的一分别浪费。
